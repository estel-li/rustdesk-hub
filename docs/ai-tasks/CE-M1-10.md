### 4.1 配置项(写入 `2fa.md` §配置项)

> 以下配置 key 与 env var 命名建议沿用 lejianwen 现有 `RUSTDESK_API_<段>_<键>` 模式;若 CE-M1-3/M1-5 实现时调整,以代码为准并同步本文档。建议命名,可调整。

| yaml key | env var | 默认值 | 说明 |
|----------|---------|--------|------|
| `mfa.enabled` | `RUSTDESK_API_MFA_ENABLED` | `false` | 全局开关,关闭时 `/api/login-mfa` 返回 404 / 503 |
| `mfa.issuer` | `RUSTDESK_API_MFA_ISSUER` | `RustDesk` | TOTP `otpauth://` URI 中的 issuer 字段 |
| `mfa.ticket-ttl` | `RUSTDESK_API_MFA_TICKET_TTL` | `5m` | 短期 ticket 有效期,范围 3–5 分钟(`docs/ai-development-plan.md` L283) |
| `mfa.recovery-code-count` | `RUSTDESK_API_MFA_RECOVERY_CODE_COUNT` | `10` | 单用户备份码数量,见 `docs/upgrade-plan.md` L97 |
| `mfa.force-group` | `RUSTDESK_API_MFA_FORCE_GROUP` | `""` | 强制启用 MFA 的组名,空字符串表示不强制 |
| `app.captcha-threshold` | `RUSTDESK_API_APP_CAPTCHA_THRESHOLD` | `3` | 已存在,MFA 错误必须复用或独立限流(`docs/ai-development-plan.md` L287-L288) |

### 4.3 API 形状(逐字写入文档,与 CE-M1-3 / M1-6 对齐)

`POST /api/login`(MFA 启用用户)响应新增可选字段:

```json
{
  "mfa_required": true,
  "ticket": "<short_jwt>"
}
```

`POST /api/login-mfa` 请求/响应:

```json
// 请求
{ "ticket": "<short_jwt>", "code": "123456" }
// 响应(沿用 /api/login 现有 token 形状)
{ "access_token": "...", "user": { "id": 1, "name": "alice", "is_admin": true } }
```

`POST /api/audit/event`(新增,沿用 Bearer 鉴权):

```json
{
  "kind": "clipboard",
  "peer_id": "123456789",
  "from_peer": "987654321",
  "from_name": "alice",
  "session_id": "uuid",
  "ip": "1.2.3.4",
  "payload_json": "{\"size\":42,\"is_file\":false}"
}
```

`GET /admin/audit_event/list?kind=clipboard&from=2026-06-01&to=2026-06-30&page=1&size=20`:返回分页列表。`/api/audit/file` 与 `/admin/audit_file/list` 保持原样(`docs/ai-development-plan.md` L341-L342)。

> 上述 JSON 仅作为运维手册示例。一旦 CE-M1-6 落地后的字段命名与此不一致,以代码 `model/audit_event.go` 与 `http/controller/api/audit.go` 为准并修订文档。

## 8. 兼容性 / 安全注意事项

- **protobuf 兼容**:本任务不改 proto,但在 2fa.md 顶部仍要提示"`/api/login` 新增字段为 optional,老客户端忽略 `mfa_required` 字段后走原流程"(对应 `docs/ai-development-plan.md` L290-L292、`docs/upgrade-plan.md` L145-L149)。
- **老客户端 ↔ 新服务端**:未启用 MFA 的用户响应必须与历史一致;启用 MFA 但客户端未升级时,文档需说明"用户应先停用 MFA、升级客户端、再启用"或"使用 Web 登录通过 OIDC/LDAP 绕开"。
- **新客户端 ↔ 老服务端**:老服务端不返回 `mfa_required`,客户端按未启用处理;手册"常见故障"小节必须列出此场景。
- **数据库迁移**:`DatabaseVersion 265 → 266`(`docs/ai-development-plan.md` L247);手册要给出回滚 SQL(DROP `user_mfa`、`audit_event` 表)与 `DatabaseVersion` 手动改回 265 的 caveat,提醒该操作仅限灾难恢复。
- **敏感字段不落盘**:`mfa_ticket` 仅内存(L51、L298-L302);备份码只存 hash(L251)。文档严禁出现明文 secret/recovery code 真实值,仅占位符演示。
- **限流**:MFA 错误纳入 captcha/ban 通道或独立 limiter(L286-L288);文档要列出 `app.captcha-threshold` 与 `app.ban-threshold` 触发后用户/管理员各自能看到什么。
- **GeoLite2/录屏审计**:audit-events.md 提到的事件 kind 必须显式排除 `record`(M3 才落地,`docs/upgrade-plan.md` L116);避免运维误以为已 GA。
- **`from_peer` 不可信**:在 audit-events.md "客户端上报" 段强调审计字段 `from_peer` 仅作展示,真实身份由服务端 token 解析(L52、L531)。

1. **## 概述** — 一段话:解释本文件覆盖的是 API/Web 账号 MFA,不是客户端会话级 2FA(`auth_2fa.rs`)。
2. **## 前置阅读** — 列表:CE-M1-1..5 任务卡链接、`docs/architecture.md` §7.4 鉴权章节、`docs/rustdesk-api.md` §鉴权章节。
3. **## 配置项** — 表格 §4.1,逐行配 1–2 句中文描述与默认值。
4. **## 启用流程** — 5–7 步:① 管理员后台开启全局 `mfa.enabled`;② 用户进入"账号设置 → MFA"扫码绑定;③ 系统下发 10 个备份码,提示用户离线保存;④ 用户下次登录拿到 `mfa_required`;⑤ 输入 TOTP → 拿 token;⑥(可选)组管理员配置 `force-group`;⑦ 审计页确认 `mfa.enable`/`mfa.consume_recovery` 事件已写入。
